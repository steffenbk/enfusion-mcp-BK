# Modding Basics — Project Setup, Data Modding, Prefabs, Entity Catalog

---

## Key File Extensions

| Extension | Type |
|---|---|
| `.et` | Entity template (prefab) |
| `.ct` | Component template (component prefab) |
| `.conf` | Config file |
| `.emat` | Material definition |
| `.edds` | Enfusion DDS texture (proprietary) |
| `.xob` | Binary model |
| `.fbx` | FBX source (converted to `.xob` on import) |
| `.c` | EnforceScript source |
| `.layout` | UI layout |
| `.ptc` | Particle system |
| `.pap` | Procedural Animation Project |
| `.siga` | Procedural Animation Signal |
| `.layer` | World layer |
| `.topo` | Topography data (2D map geometry) |
| `.nmn` | Navmesh |
| `.meta` | Asset metadata + GUID (every registered asset has one) |
| `.gproj` | Project file |
| `.st` | Localisation string table |

**Critical:** Engine identifies assets by GUID only (stored in `.meta`). File name and path are irrelevant for overrides/replacements.

**Pitfall:** `Prefabs/Auto/` is auto-generated by Workbench plugins. Do not manually edit.

## Mod Project Setup

**Prerequisites**: Arma Reforger + Workbench installed.

**Add ArmaReforger project to Workbench**:
1. Start Workbench via Steam > Tools > Arma Reforger Tools > START, or double-click `ArmaReforgerWorkbenchSteam.exe`.
2. Click "Add Existing" → browse to `<Arma Reforger install dir>\addons\data\ArmaReforger.gproj`.

**Create new mod**:
1. Click "Create New" in Launcher.
2. Name: letters, numbers, spaces, `-`, `_`, `.` only.
3. Location: default `%userProfile%\Documents\My Games\ArmaReforgerWorkbench\addons`.
4. Pick dependencies (ArmaReforger is default and required).
5. Click OK.

**Rules**:
- Do NOT create projects in OneDrive directories — they fail to load.
- Location must have write permissions (e.g., `C:\Users\<user>\Documents`, not `C:\Program Files`).
- Dependencies must be visible in project list (including dependencies-of-dependencies).
- `addon.gproj` can be renamed; re-add to project list after rename.
- Multiple projects can be launched together via "Open with Addons" context menu option.

**Troubleshooting**:
- Missing dependency error in log: `ENGINE (E): Addon 'X' dependency 'GUID' can't be added` → add the dependency mod to project list.
- Empty dependency `''` in `.gproj`: manually edit `.gproj` in text editor, remove the empty string from Dependencies array.
- Project read-only (OneDrive/GDrive sync): run batch script in addon root to remove read-only attributes.

## Data Modding: Override / Duplicate / Inherit / Replace

### Moddability by File Type

| Extension | Replace | Modify (Override) | Inherit |
|---|---|---|---|
| `.et` (Prefab) | — | Yes | Yes |
| `.conf` (Config) | — | Yes | Yes |
| `.xob` (Model) | Yes | — | — |
| `.emat` (Material) | — | Yes | Yes |
| `.edds` (Texture) | Yes | — | — |
| `.c` (Script) | Yes [same path] | `modded` keyword | — |
| `.layout` | — | Yes | Yes |
| `.ptc` (Particles) | Yes (via Create Variant) | — | — |
| `.ent` (World) | Yes | — | — |
| `.bt` (Behavior Tree) | Partial (attempts override) | — | — |
| `.anm`, `.agf`, `.txa`, etc. | Yes | — | — |

**Notes**:
1. Prefabs, configs, layouts are always in override mode — cannot be fully replaced.
2. Script replacement: place same file at same relative path in your addon.
3. `modded` keyword required for script class modification; method must not be private.

### Override
Creates a file with the **same GUID** as the original. Only changes are stored in your addon; rest kept from original.
- Context menu: "Override in '[addon]'" — only for prefabs, layouts, configs.
- Engine identifies assets by GUID only — name and location don't need to match.

### Replacement (full)
Completely replaces original. For models, textures, sounds, animations, particles, world files, behavior trees.
- Obtain original GUID: right-click → Copy Resource GUID(s).
- Manually set the GUID in your new file's meta file to match original.

### Duplicate
Creates a file with a **new unique GUID**. Contains all original data. Does not affect original.
- Context menu: "Duplicate in '[addon]'"

### Inherit
Creates a child file that inherits all data from parent. Has its own unique GUID.
- Context menu: "Inherit in '[addon]'" or "Create Inherited File"
- Prefab inheritance only available via Resource Browser attached to World Editor.

### Navigation
- "Navigate to Original" — jumps to original unoverridden file.
- "Navigate to Override" — jumps to first override in chain (repeatable through chain).
- "Navigate to Ancestor" — jumps to first ancestor of overridden file.

### Script Modding Pattern
```
// Extend class (no new parent = just adding/overriding methods):
modded class SCR_PlayerScoreInfo {}

// Override method:
modded class SCR_ScoringSystemComponent : SCR_BaseGameModeComponent
{
    override void AddSuicide(int playerID)
    {
        super.AddSuicide(playerID);  // call original
        // Your code
    }
}
```

## LOD System

- LOD0 = highest quality; higher numbers = lower quality (rendered at greater distance).
- `_LODn` suffix in FBX object name assigns mesh to LOD n.
- **LOD0→1 switch** factors: spherical size, triangle count of LOD0, FOV, graphics settings.
- **LOD1→2→3+ switch**: each LOD switches when its screen surface size is ~half vs initial appearance AND relative triangle count between LODs is ~half.
- **Debug**: Alt+Win → Scene/Colorize objects LODs in World Editor (each LOD has a color overlay).

**Manual override — LOD Factors** (on `MeshObject` component, `LOD Factors` array):
- One factor per LOD. Auto-fills based on XOB LOD count.
- Factor multiplies effective triangle count for switch distance. >1 = switch sooner, <1 = switch later.
- Factors inherited from parent prefabs.
- Save: switch to specific inherited prefab in Object Properties, set factor, save world.

## Game Mode Setup

**Required entities** (single instance each):

| Entity | Pre-made | Base Prefab |
|---|---|---|
| `SCR_BaseGameMode` | `GameMode_Plain.et` | `GameMode_Base.et` |
| `SCR_FactionManager` | `FactionManager_FFA.et` / `FactionManager_USxUSSR.et` | `FactionManager_Base.et` |
| `SCR_LoadoutManager` | `LoadoutManager_FFA.et` / `LoadoutManager_USxUSSR.et` | `LoadoutManager_Base.et` |
| `SCR_SpawnPoint` | `SpawnPoint_FFA.et` / faction-specific | `SpawnPoint_Base.et` |

**Key components**:
- `SCR_RespawnSystemComponent` — on game mode entity; mandatory.
- `SCR_RespawnComponent` — on player controller. Client gets own; authority iterates all.

**Accessing game mode API**:
```
SCR_BaseGameMode gameMode = SCR_BaseGameMode.Cast(GetGame().GetGameMode());
SCR_FactionManager factionManager = SCR_FactionManager.Cast(GetGame().GetFactionManager());
SCR_LoadoutManager loadoutManager = GetGame().GetLoadoutManager();
SCR_RespawnSystemComponent respawnSystem = gameMode.GetRespawnSystem();
```

**Game states**: `PREGAME`, `GAME`, `POSTGAME`. Transition to POSTGAME: `gameMode.EndGameMode(SCR_GameModeEndData...)`.

**Extend game mode via** `SCR_BaseGameModeComponent` subclasses — many work standalone and are mix-and-match.

## 2D Map Creation

1. Place `MapEntity.et` (`$ArmaReforger:Prefabs/World/Game/MapEntity.et`) in world.
2. Fill **Map Geometry Data** with `.topo` file.
3. Fill **Satellite Background Image** with `.edds` texture.

**Generate .topo**: WE → Export Map Data → Type: Geometry2D → Export (slow).

**Generate satellite background**:
1. WE → Export Map Data → Type: Rasterization → Export → 4096×4096 upside-down `.tga`.
2. Flip vertically in Paint.NET or GIMP → save as PNG.
3. Place in `Data\UI\Textures\Map\worlds\`.
4. Register in Resource Manager → Import Settings → Color Space: `ToSRGB` → Import.

## Prefabs — Operations

### Terminology
- **Prefab** (`.et`) — entity template with entities, components, configs.
- **Component Prefab** (`.ct`) — components only.
- **Config Object Prefab** (`.conf`) — a sub-object config.
- **Prefab root** — top-level entity in prefab hierarchy.
- **Prefab instance** — entities created from a prefab in the scene.
- **Prefab Edit Mode** — WE mode for editing a prefab directly (Save = Ctrl+S, saves to prefab not world).

### Creating Prefabs
- Drag entity root from Hierarchy window to Resource Browser → enter name.
- Drag existing `.et` file from Resource Browser into WE viewport → creates inherited prefab instance.
- Drag `.xob` model from Resource Browser into WE → creates `GenericEntity` + `MeshObject` component.

### Creating Component Prefabs
- Drag component from Object Properties window to Resource Browser → enter name.

### Adding/Removing Components
- Add component: Object Properties → Add component button.
- Add child component: right-click component → Add child component.
- Remove: right-click → Delete Component.
- From prefab (not instance): switch inheritance tree to prefab, then delete.

### Editing Prefab Properties
**Method 1 — Inheritance tree**: Switch combo box from "Entity instance" to the prefab name. Changes affect all instances immediately.

**Method 2 — Apply to prefab**: Edit on instance, then click "Apply to prefab" (bottom of Object Properties). Can choose which parent prefab to apply to.

**Single property**: Right-click property → "Apply value to prefab".

**Warning**: Changes to prefabs are permanently stored only after saving the world (Ctrl+S or Ctrl+Shift+S).

### Adding Entities to Prefab
1. Create instance near existing prefab instance.
2. Position/orient as desired.
3. Hold **Alt**, drag entity in Hierarchy window and drop into prefab hierarchy.
4. Click Apply to Prefab.

### Removing Entities from Prefab
- Right-click entity in hierarchy → "Delete from prefab". (Del key only removes instance.)

### Instancing Prefabs
- Drag `.et` from Resource Browser to viewport.
- Select in Resource Browser + hover over scene + **F1** — places at cursor (good for bulk placement).
- Ctrl+C / Ctrl+V on existing instance.

### Identifying Prefab Membership
- Prefab column in Hierarchy window shows prefab names.
- Context menu: "Select prefab instance root", "Select prefab instance members", "Show prefab in resource manager".
- **Bold property name** in Object Properties = property is overridden.

### Breaking Prefab Instance
Right-click → "Break prefab instance" — converts to normal entities, removes prefab links.

### Prefab Edit Mode
Open via: Resource Manager tab → Edit prefab button, or RMB on `.et` in Resource Browser → "Edit prefab(s)".
- Loads empty terrain in WE — ensure no unsaved changes in current world first.
- Can open multiple prefabs at once.

## Entity Catalog

**Purpose**: Cohesive list of faction/factionless prefabs used by multiple systems. Avoids needing to spawn prefabs to query their data.

**Faction catalogs**: In faction `.conf` files (e.g., `Configs/Factions/US.conf`). Edit via the Catalog config, not directly in the faction config.

**Factionless catalog**: On `SCR_EntityCatalogManagerComponent` on `GameMode_Base.et`.

### Catalog Classes
- `SCR_EntityCatalog` — single flat list.
- `SCR_EntityCatalogMultiList` — single list + sub-lists (e.g., ITEM catalog with WEAPONS sub-list). Sub-lists merged into full list on init.

### Entry Classes
| Class | Use |
|---|---|
| `SCR_EntityCatalogEntry` | Editable entity prefab (auto-gets UIInfo/Labels from `SCR_EditableEntityComponentClass`) |
| `SCR_EntityCatalogEntryNonEditable` | Base for non-editable — do NOT use directly |
| `SCR_EntityCatalogEntryCustomInfo` | Non-editable entity with manual UIInfo + Labels |
| `SCR_EntityCatalogInventoryItem` | Inventory items (UIInfo from InventoryItemComponent not accessible) |

### Entry Fields
- `Prefab` — ResourceName of the prefab.
- `Enabled` — if false, entry removed on init (use for WIP prefabs or to remove entries from mods).
- `Labels` — `EEditableEntityLabel` labels for filtering.
- `UI Info` — Name, Description, Icon (auto from EditableEntityComponent or manual).
- `Entity Data List` — system-specific data objects attached to entry.

### Entry Data
Each entry has a data list. Data objects are "components" of the entry. Examples:
- `SCR_ArsenalItem` — arsenal item type/mode.
- `SCR_EntityCatalogSpawnerData` — supply cost, spawn slots.

**Filtering API**:
```
catalog.GetEntityListWithData<SCR_EntityCatalogSpawnerData>(list);
// Also: GetEntityList*() variants to filter by labels
```

**Key advantage**: Systems get entity data without spawning the prefab; modders can add data to entries without breaking other systems.
